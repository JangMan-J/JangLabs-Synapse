# Pending spec seeds — distilled from the removed `.planning/` tree

These are **current-state capability facts** about the synapse memory subsystem, distilled
from the GSD `.planning/` tree before it was removed. The raw artifacts they were drawn
from are recoverable byte-for-byte at git tag **`gsd-archive-pre-removal`** — nothing here
is lost, only relocated out of the live tree.

**This is a seed list, not a backfill.** Per the grilled Q4 policy, `openspec/specs/` grows
**organically**: a real `openspec/specs/<capability>/spec.md` entry is written when that
capability is **next touched**, not preemptively now. Each bullet below is a one-line
reminder of a capability that already exists in the live engine (`lib/memory_surface.py`)
and its hooks, so that the organic spec — when it is eventually authored — starts from the
distilled truth rather than re-deriving it. Promote a bullet into a real spec entry as you
touch its capability; delete it from this list once promoted. Do **not** treat the presence
of a bullet here as an obligation to write its spec now.

Source paths are given relative to the lab; `.planning/...` paths resolve only at the
archive tag, engine/`memory/` paths are live.

## Capability seeds

- **collision-projection**: `project_triggers(memdir, triggers, stem=None)` projects a proposed trigger set against the on-disk catalog and returns `{collisions, distinct_count, per_trigger}` — the distinct OTHER existing memories the set would co-fire with — by reusing the read-path matcher via the shared `_walk_index` helper (no second matcher) but WITHOUT the `_meets_min_candidate` surface gate, so single-weak-tier co-fires are reported. `per_trigger` gives each raw pattern's distinct match count (including 0-count) so "the whole set is noise" is distinguishable from "one trigger is broad but the set discriminates"; the proposed memory's own `stem` is excluded from its hits; reads the compiled catalog only (never rebuilds, never mutates `memory/`); fails open — any internal error returns a FRESH `{collisions:[], distinct_count:0, per_trigger:{}}` and never raises. (src: lib/memory_surface.py, .planning/phases/05-collision-projection-engine/05-01-SUMMARY.md)

- **memory-grammar**: `memory/_grammar.md` is the single unified trigger grammar — each tag entry declares a one-line gloss, a placement hint (box|project|either), and evidence fields (commands/paths/args/synonyms/related). `validate-grammar`/`validate_grammar` enforces that every tag declares at least one behavioral evidence pattern (a synonyms-only tag fails with exit 2; a tag without observable triggers cannot exist). Parsed by a stdlib-`re` scanner (not PyYAML/JSON), lives in the lab, relative-symlinked into the box-brain store; it superseded legacy `_tags.md`/`_tag_links.md` as the sole routing source at the Phase-2 flip. (src: memory/_grammar.md, .planning/milestones/v1.0-phases/01-trigger-grammar-write-time-intelligence/01-01-SUMMARY.md)
- **write-time-triggers**: Saving a memory derives its trigger patterns in-context at write time. `memory-write-context.sh` injects (via engine `write-context`) one composite under a 9500-char budget — trigger-spec schema + worked example, grammar vocabulary (or a one-line-per-tag digest fallback if over budget), top-N dedup candidates, and placement guidance. The `triggers:` block is stored nested under `metadata:` in frontmatter (commands/paths/args/synonyms string arrays); a top-level `triggers:` is rejected for parity with top-level `tags:`. The write-guard runs engine `check-write`, validating trigger shape and a specificity gate (rejects sets that are only generic verbs or only broad globs like `**` / `~/**`). (src: lib/memory_surface.py, .planning/milestones/v1.0-phases/01-trigger-grammar-write-time-intelligence/01-02-SUMMARY.md)
- **write-dedup**: New memory writes are deduplicated against the store before triggers are authored. Similarity from the catalog is `score = 0.6*tag_overlap + 0.4*bag-of-words-cosine` (stdlib `collections.Counter` only). Layer 1 (advisory): write-context injects the top-N most-similar existing memories so the in-context model consolidates instead of duplicating. Layer 2 (backstop, fail-closed): `check-write` denies a new-file write whose best score >= `DEDUP_BACKSTOP_THRESHOLD` (0.85), naming the existing file's path. The backstop fires only on new files; writing into an existing file (consolidation) is always allowed. (src: lib/memory_surface.py, .planning/milestones/v1.0-phases/01-trigger-grammar-write-time-intelligence/01-03-SUMMARY.md)
- **write-placement-gate**: Memory writes route to the correct store by subject. Detection spans the box-brain store, any Claude project store (`*/.claude/projects/*/memory/*.md`), and repo `memory/` dirs (`*/memory/*.md`), with infra files (underscore-prefixed, `MEMORY.md`, `_grammar.md`) exempt as the first check. `_classify_target` returns `box | project-store | repo-memory | other` via realpath-normalized prefix comparison (prevents `../` escape). Enforcement is graduated: guidance is always injected, but the guard DENIES only high-confidence misplacement — a memory all of whose grammar-known tags carry `placement: box` written to a non-box target — with the correct box-store path in the deny reason; ambiguous/mixed placement fails open. (src: lib/memory_surface.py, .planning/milestones/v1.0-phases/01-trigger-grammar-write-time-intelligence/01-04-SUMMARY.md)
- **routing-index**: The routing index is a pure build artifact. `rebuild()` compiles a `triggerIndex` section into `_memory_catalog.json` from grammar tag-level evidence plus per-memory `triggers:` frontmatter (folded into shared `byCommand`/`byPath`/`byArg`/`bySynonym` buckets, with `byMemoryId` for direct/derived routing). One command (`engine rebuild`) reconstructs it fully from store contents; it is never hand-edited. `rebuild()` emits a `routabilityReport` (count + ids of unroutable memories) on stderr and in catalog metadata. Consistency is structural: every engine mutator (`add_tag`/`link`/`unlink` via `_mutate_then_validate`) calls `rebuild()` before returning, and PostToolUse `memory-catalog-refresh.sh` rebuilds on any store `.md` write (grammar/taxonomy writes via the lab-side symlink are resolved through `readlink -f`). `fingerprint()` hashes `_grammar.md` so staleness is detectable. Ranking-metadata writes (`lastReviewed`/`declineCount`) intentionally do NOT rebuild — they are ranking inputs, not routing inputs. (src: lib/memory_surface.py, .planning/milestones/v1.0-phases/02-routing-index-live-recall-cutover/02-01-SUMMARY.md)
- **memory-recall**: Per-tool-call recall is a precomputed inverted-index lookup over behavioral evidence extracted from `tool_input` (command basenames, canonicalized paths, arg tokens) — no LLM, no embeddings. A single matcher (`search()`) walks both levels (grammar-tag evidence and per-memory triggers) over the catalog's `triggerIndex`/`recallVocab` only; it never reads `parse_tags_md`/`parse_tag_links` (write-path only after the flip). Scoring uses `TIER_WEIGHTS = {strong:10, medium:6, weak:3}` (command/path strong, arg medium, synonym weak). Surface gate is >=1 strong-tier tuple OR >=2 total tuples; silence is the default and no-evidence calls emit nothing. Every surfaced memory cites its firing evidence inline as `{tag} <- {trigger_type}:{matched_value}` (e.g. `why: nvidia <- command:nvidia-smi`) so a wrong fire is diagnosable from the recall block alone. (src: lib/memory_surface.py, .planning/milestones/v1.0-phases/02-routing-index-live-recall-cutover/02-02-SUMMARY.md)
- **memory-recall-read-path-invariant**: HARD read-path invariant: `search()` must NEVER rebuild and NEVER load memory bodies. A missing/corrupt catalog fails CLOSED (returns None, surfaces nothing) — NOT because recall is non-advisory, but because calling `rebuild()` inside `search()` would read memory frontmatter/bodies during a read-path call, violating the bodies-never-loaded constraint (engine §19). This is the single non-obvious rule most likely to be violated by a future "just rebuild if stale" read-path change (called out as the top anti-pattern in the Phase-2 research). At the hook level the call still fails OPEN (exit 0, tool call never blocked). (src: lib/memory_surface.py:1878-1879 & :2192, .planning/milestones/v1.0-phases/02-routing-index-live-recall-cutover/02-RESEARCH.md)
- **fail-open-kill-switch**: Whole-surface fail-open / break-glass contract for the memory hooks: (1) a `.surface-disabled` file in the store is a kill-switch that makes EVERY memory hook exit 0 with empty stdout/stderr for any tool call (recall pipeline fully suppressed); (2) with `_memory_catalog.json` MISSING the recall hook exits 0 and NEVER rc 2 — it fails open, not hard; (3) recall is advisory-only, silence-is-default, and a missing/unreadable engine never blocks the tool call. These are verified-live present-tense capabilities, independent of any planning-system spine. (src: .planning/milestones/v1.0-MVR.md, lib/memory_surface.py)
- **path-canonicalization**: Hook/engine path canonicalization deliberately DIVERGES and the rule is load-bearing: the shell hooks canonicalize LEXICALLY with `realpath -sm` (NOT resolving symlinks) because the store's taxonomy infra files (`_tags.md`/`_tag_links.md`/`_grammar.md`) ARE relative symlinks into `synapse/memory/` and resolving them would break store-path gating; the engine resolves symlinks via `os.path.realpath`/`os.path.expanduser`. A target can be classified one way by the hook and differently by the engine — this is intentional and any edit touching store-path classification must preserve it. (src: .planning/milestones/v1.0-phases/01-trigger-grammar-write-time-intelligence/01-PATTERNS.md, hooks/*.sh)
- **telemetry-capture**: Every recall fire appends one JSONL record `{ts, qid, mems:[{id,tag,type,val}], conf}` to `_recall_telemetry.jsonl` AFTER the advisory emission and fail-open (`|| true`) so a telemetry fault never blocks recall. The file is size-gated: at >=1MB it rotates to `_recall_telemetry.jsonl.1` (one generation) via atomic `mv`. A separate read-confirmation record `{ts, id, signal:"read"}` is appended by `memory-catalog-refresh.sh` when a Read targets a store memory file with a live (<15min) dedup mark — the mark presence IS the fire↔read correlation (no timestamp join). Fire-append is gated on at least one dedup mark having persisted, so unloggable-read situations produce zero-fire (never demoted) rather than fires-without-possible-reads. (src: lib/memory_surface.py, .planning/milestones/v1.0-phases/03-telemetry-self-curation/03-VERIFICATION.md)
- **automated-maintenance-pass**: `maintenance()` is the curation pass that replaced Memory Roulette. Triggered from `memory-base-floor.sh` at SessionStart only when `_recall_telemetry.jsonl` has grown >=50 records since the last pass (tracked in `_maintenance_state.json`), runs under `timeout 2 || true` so it never blocks session start, and serializes on an O_EXCL lock with atomic-rename (not unlink) stale-reclaim. It scores each memory over a rectangular `telemetryWindowDays` window: `read_rate >= promoteThreshold (0.4)` clears `declineCount`, `read_rate <= demoteThreshold (0.05)` increments it; thresholds live in `_memory_surface_config.json`. It NEVER deletes/moves/rewrites memory content — only frontmatter `declineCount` via `parse_frontmatter`→`generate_frontmatter`→`write_atomic`. Two floors prevent runaway decay: the D-43 zero-fire floor (`fire_count==0` → never demoted) and the minimum-evidence guard (no real mutations until >=10 distinct session-days OR >=30 days span; otherwise `insufficient_evidence` and defer). NON-IDEMPOTENCY invariant: the pass must CLAIM state (`_update_maintenance_state`) BEFORE applying any `declineCount` mutation, because per-file writes are atomic but the PASS is not — a mid-loop failure or the `timeout 2` SIGTERM (which skips Python's `finally`) would otherwise replay the pass and re-increment already-demoted memories. A `maintenance-shadow` subcommand computes the same lists without writing; a one-line summary is injected into the SessionStart base-floor block. (src: lib/memory_surface.py, .planning/milestones/v1.0-phases/03-telemetry-self-curation/03-VERIFICATION.md, 03-REVIEW-FIX.md)
- **seat-governance**: `MEMORY.md` router-seat membership is machine-governed inside the maintenance pass (`seats()`), never by hand-audit. A seat is proposed for demotion only when BOTH a probe payload demonstrably surfaces that memory through the live recall hook (`covered:true` in `_seat_probe_results.json`, written by `seat_probes.py`) AND telemetry shows it fired in real sessions — under the same minimum-evidence window. Because the box-brain store is NOT git-tracked, changes are emitted as a human-vetoable `PENDING-SEAT-CHANGES` HTML-comment block prepended to `MEMORY.md` (delete the block to approve); non-block content stays byte-identical and re-runs replace rather than stack. With current always-on seats carrying no `triggers:` block, all probes correctly return `covered:false`/no-derivable-probe, so the engine proposes zero demotions — the absence of a derivable probe is itself proof the seat belongs in the router. (src: lib/memory_surface.py, .planning/milestones/v1.0-phases/03-telemetry-self-curation/03-SEAT-GOVERNANCE.md)
- **harness-lint**: `scripts/lint.sh` is a manual, operator-invoked shellcheck runner (NOT a registered hook) over the harness shell scripts: globs `hooks/*.sh` plus itself plus `fix-memory-plug.sh` (if present), resolves the repo root from `${BASH_SOURCE[0]}` so cwd is irrelevant, accepts an optional `--severity=LEVEL` passthrough, and propagates shellcheck's aggregate exit code. It deliberately INVERTS the two harness hook rules: it prints on success (`shellcheck: N files clean`) and fails CLOSED (exit 127 with an actionable `paru -S shellcheck` / `sudo pacman -S shellcheck` line) when shellcheck is absent. It keeps the per-edit hot path (`hooks/syntax-check-touched.sh`) at `bash -n` only. (src: scripts/lint.sh, .planning/quick/260616-1pm-add-two-harness-automations-scripts-lint/260616-1pm-PLAN.md)
- **hook-reviewer**: `.claude/agents/hook-reviewer.md` is a read-only, advisory subagent (tools: Read, Grep, Glob, Bash — explicitly NO Edit/Write) that audits CHANGED `hooks/*.sh` (scoped via `git diff` against the merge-base) against the four harness iron rules: quiet-on-success, fails-open, never-writes-permissions, cheap (pure POSIX-ish shell + jq). It recognizes the sanctioned exceptions (`memory-recall.sh` is the sole per-call Python-spawning hook; operator-CLIs like `lint.sh` are allowed to fail-closed and are out of scope) and emits findings as `file:line - invariant - what` entries; it never edits. Committed in-repo, NOT registered in `settings.global.fragment.json` (it is an agent, not a hook); it required `git add -f` to override the global `.claude/` gitignore rule. (src: .claude/agents/hook-reviewer.md, .planning/quick/260616-1pm-add-two-harness-automations-scripts-lint/260616-1pm-PLAN.md)
