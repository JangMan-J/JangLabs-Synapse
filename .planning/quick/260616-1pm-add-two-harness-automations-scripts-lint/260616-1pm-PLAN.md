  <action>
Create the directory `scripts/` and the file `scripts/lint.sh` — a manual/opt-in shellcheck
runner for the harness shell scripts. This is NOT a hook; it is an operator-invoked CLI.

Script requirements (write the file with these exact behaviors):
- Shebang `#!/usr/bin/env bash` and `set -euo pipefail` on the second line (committed scripts use bash, per lab convention).
- A leading comment block stating: (a) this is a MANUAL/opt-in lint runner, NOT a hook, so the
  harness "quiet on success" rule does NOT apply — output-on-success is EXPECTED here; (b) the
  usual harness "fail open" rule is INVERTED here because this is an operator-invoked CLI, not a
  tool-call hook — a missing `shellcheck` is an actionable error the operator can fix, so the
  script fails CLOSED (exits non-zero) rather than silently. Note both inversions so a future
  reader does not "fix" them.
- Resolve the repo root robustly: cd to the directory of the script's own dir's parent
  (e.g. via `dirname` of `${BASH_SOURCE[0]}` then `..`), so the script works regardless of cwd.
- Missing-dependency branch FIRST: if `command -v shellcheck` fails, print ONE actionable line
  to stderr telling the operator to install it: `paru -S shellcheck` or `sudo pacman -S shellcheck`
  (it is in the `extra` repo on this Arch/CachyOS box), then exit non-zero (e.g. exit 127 or exit 1).
- Optional `--severity=LEVEL` passthrough (single optional arg). Default = shellcheck's default
  (warning). Keep it simple; do NOT add `-x` (follow-sourced-files) — not needed here.
- Build the target file list robustly (space-safe, even though there are no spaces here): the
  glob `hooks/*.sh`, plus `scripts/lint.sh` itself, plus `fix-memory-plug.sh` (the break-glass
  script at repo root) ONLY if it exists. Use a bash array and a nullglob-safe approach (e.g.
  `shopt -s nullglob` for the `hooks/*.sh` glob, append the others conditionally), then invoke
  shellcheck once over the array: `shellcheck [--severity=LEVEL] "${files[@]}"`.
- Propagate shellcheck's exit code: capture it, and exit with it (aggregate — if any file fails,
  the overall exit is non-zero). On success, print a terse one-line summary to stdout, e.g.
  `shellcheck: N files clean` where N is the file count. On failure, shellcheck's own findings go
  to stdout/stderr as usual and the script exits with shellcheck's non-zero code.
- Do NOT place fenced code in this file beyond the script itself; the script IS the deliverable.

DO NOT register this in settings.global.fragment.json. DO NOT modify any hook.

After writing the file:
1. Run `chmod +x scripts/lint.sh`.
2. Verify syntax without executing the lint body: `bash -n scripts/lint.sh` must be clean.
3. RUN it to demonstrate the missing-shellcheck branch (shellcheck was NOT installed at last
   check): `bash scripts/lint.sh; echo "exit=$?"`. Record the VERBATIM output (the actionable
   install line + non-zero exit) for the SUMMARY. If shellcheck IS present on this box, run it
   for real instead and record that verbatim output; do NOT auto-install shellcheck via sudo —
   if absent, note that installing it is the operator's call.
4. Stage with an EXPLICIT pathspec and the exec bit, then VERIFY the mode before committing:
   `git add scripts/lint.sh` then `git ls-files -s scripts/lint.sh` MUST show mode `100755`.
   (Known failure mode: a script committed 100644 silently breaks on a later checkout.)
5. Commit ONLY this file: `git commit scripts/lint.sh -m "..."` (or `git commit` after the
   single explicit `git add`). NEVER `git add -A` / `git add .`.

this lab repo. Scope: HARNESS-INVARIANT FOCUSED (narrow; complements, does not duplicate, the
existing `gsd-code-reviewer`). It reviews touched `hooks/*.sh` against the 4 harness iron rules.

Model the YAML frontmatter shape on the existing `~/.claude/agents/gsd-code-reviewer.md`
(`name:` / `description:` / `tools:`) but keep the content lab-specific and narrower.

YAML frontmatter requirements:
- `name: hook-reviewer`
- `description:` a single line that TRIGGERS on reviewing/auditing touched harness hook scripts
  (e.g. "Read-only reviewer that audits touched synapse harness hooks/*.sh against the 4 harness
  iron rules (quiet-on-success, fails-open, never-writes-permissions, cheap). Advisory only —
  reports findings, never edits.").
- `tools: Read, Grep, Glob, Bash` — a READ-ONLY set. Bash is included ONLY for read-only
  inspection (`git diff`, `git ls-files`, `git merge-base`). EXPLICITLY NO `Edit` and NO `Write`.
- Optionally a `color:` and `effort:` line matching the gsd-* shape (keep minimal).

Body requirements (Markdown):
- Open with a one-line statement that this agent is ADVISORY and READ-ONLY: it reports findings,
  never edits any file.
- Scoping instructions: scope to SHELL HOOK FILES under `hooks/` that CHANGED. Determine the
  changed set via `git diff` against the merge-base (e.g. `git merge-base HEAD origin/main` or a
  range passed in the prompt), filtered to `hooks/*.sh`. SKIP non-hook files and unchanged files.
- The 4 invariants as a checklist. For EACH: (a) what to look for, (b) the known sanctioned
  exception(s) the reviewer should NOT flag, (c) cite findings as `file:line`.
  1. **Quiet on success** — what to look for: any stdout on the success path, or `[ok]`/`[skip]`/
     progress lines; a hook must exit 0 with NO stdout on success, stderr reserved for genuinely
     actionable failure only. (No special exception.)
  2. **Fails open** — what to look for: a missing engine/dep, unreadable input, or unexpected
     error that does NOT exit 0 / that blocks the tool call; only genuinely actionable taxonomy/
     config errors may exit 2. Sanctioned exception (do NOT flag): operator-invoked CLIs like
     `scripts/lint.sh` are ALLOWED to fail-closed — but those are not hooks, so they are out of
     this reviewer's `hooks/*.sh` scope anyway; note this so the reviewer never reaches for it.
  3. **Never writes permissions** — what to look for: ANY write to a `permissions` key in any
     settings.json (not `allow`/`deny`, not `defaultMode`, not `disableAllHooks`/bypass flags).
     Flag any such write. (No exception.)
  4. **Cheap** — what to look for: heavy subprocess use, a per-call Python interpreter spawn
     (`python3 -c ...`), or other non-cheap work on the per-tool-call path; hooks should be pure
     POSIX-ish shell + jq. Sanctioned exception (do NOT flag): `memory-recall.sh` is the SOLE
     sanctioned per-call Python-spawning hook (its startup cost is amortized under the ≤55ms p95
     budget) — the reviewer should RECOGNIZE it and not flag it.
- A short "What this is NOT" note: NOT a general bug/security reviewer (that is `gsd-code-reviewer`);
  NOT a style linter (that is `scripts/lint.sh` / shellcheck).
- An output-format note: emit findings as a terse list of `file:line — <invariant> — <what>`
  entries, classified advisory; if no findings, say so plainly. The agent reports; it never edits.

DO NOT add this file to `settings.global.fragment.json` (it is an agent, not a hook). DO NOT run
`agent-harness.py install`.

2. Stage with an EXPLICIT pathspec: `git add .claude/agents/hook-reviewer.md`. NEVER `git add -A`/`git add .`.
3. Commit ONLY this file in its own atomic commit.
