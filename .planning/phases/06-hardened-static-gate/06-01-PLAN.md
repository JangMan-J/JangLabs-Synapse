<objective>
Harden the existing blocking write-time trigger gate (`_check_triggers` in
`lib/memory_surface.py`) so a trigger set whose ONLY behavioral evidence is a real-but-broad
LOW-SIGNAL command (bare `git`, `cat`, `ls`, `cd`, `python3`, `bash`, …) — with no narrowing
arg and no specific (non-broad) path — is DENIED at write time, exactly as generic verbs are
denied today. Pairing such a command with a narrowing arg or a specific path PASSES.

This is the static, corpus-free half of the v1.1 noise fix (design spec Component 2). It is
ADDITIVE: the existing generic-verb-only and broad-glob-only deny arms — and their tests —
stay green unchanged. The change is a new `LOW_SIGNAL_COMMANDS` vocabulary set (D-01), a
one-predicate broadening of the existing specificity-gate deny block (D-02), and a refined
deny message. No corpus lookup, no projection, no hook edits this phase.

Purpose: A memory written with `triggers: {commands: [git]}` (the observed bare-`git`
over-fire — three git-tagged memories co-firing at high confidence on the bare `git` token)
is the precise noise failure mode the design philosophy warns against. Phase 6 stops it at
birth, statically.

Output:
- `lib/memory_surface.py`: `LOW_SIGNAL_COMMANDS` module-level set beside `GENERIC_VERBS`;
  broadened deny predicate in `_check_triggers`; refined deny message naming the offending
  command(s).
- `tests/memory_surface/test_write_triggers.py`: explicit fixtures pinning the contract
  (GATE-01/GATE-02/QC-02) plus guards that the existing deny arms still fire (D-04).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| memory author -> write-guard | A memory file being Written/Edited carries an author-supplied `triggers:` block; `_check_triggers` is the blocking validator at this boundary. This phase tightens it (more author inputs are now correctly DENIED). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | `_check_triggers` deny predicate | accept | A too-narrow predicate lets a degenerate bare-`git` trigger through (status-quo noise); a too-broad one falsely denies a legitimate single-command memory. Mitigated by the D-01 "when in doubt leave it OUT" membership bar and explicit pass/deny fixtures (Task 1) pinning both directions. |
| T-06-02 | Denial of Service | `_check_triggers` (BLOCKING gate) | accept | This validator is intentionally blocking (rc 2 = deny); per 06-CONTEXT / synapse CLAUDE.md it is the one place denial is correct, so fail-open is explicitly NOT applied here. A wrongly-denied legitimate write is recoverable — the message tells the author to add a distinguishing arg/path. |
| T-06-03 | Information Disclosure | deny message | accept | The deny message names only the author's own offending command(s) back to them; no corpus or other-memory content is read or disclosed (this phase is corpus-free — no `project_triggers`, no catalog read). |
| T-06-SC | Tampering | npm/pip/cargo installs | n/a | No package installs this phase — stdlib-only, no new dependencies. No legitimacy gate required; T-06-SC carries no action. |
</threat_model>
