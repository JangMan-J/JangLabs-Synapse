## Phase Goal

**As a** solo operator of this box's memory harness, **I want to** have every recall fire logged, every read-confirmation detected, and curation decided automatically from that evidence, **so that** the right memories keep surfacing with zero human curation ritual — Roulette retired and router seats machine-governed.

This plan delivers the first vertical slice: after it completes, real fire events and real read-confirmations from live sessions land in `_recall_telemetry.jsonl` — the user can `tail` the file and watch the system observe itself, and the recall path provably still meets its ≤55ms p95 budget.

<objective>
Capture the telemetry that every downstream curation decision feeds on: extend the live recall hook to append a fire event per emission (D-33/D-34/D-35/D-36), extend the catalog-refresh hook with a Read-signal arm correlated via dedup marks (D-37/D-38), and register + activate the Read matcher in the settings fragment.

Purpose: CUR-01 (append-only bounded fire log) and CUR-02 (read-confirmation signal) — the data layer for CUR-03/04/05.
Output: telemetry-appending `memory-recall.sh`, Read-arm `memory-catalog-refresh.sh`, Read matcher in fragment + live settings, contract tests, and a post-ship bench record proving the p95 budget holds.
</objective>
