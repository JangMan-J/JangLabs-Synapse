## Phase Goal

**As a** Claude session making tool calls on this box, **I want to** have the right memories surface from a near-free precomputed evidence lookup that explains every fire, **so that** recall is trustworthy, diagnosable in seconds, and the legacy routing path can be removed with every existing memory still reachable.

This plan delivers the third vertical slice: after it completes, the user can run one command that PROVES both probe directions through the real hook with visible tuples, and one command that prints the real p95 against the ≤50ms gate — the two remaining MVR demonstrations that need new tooling.

<objective>
Build the probe runner and benchmark harness (D-32, CORE-09) that drive the REAL `memory-recall.sh` — past bugs lived in the shell layer, so engine-only tests don't count — and close the performance budget to ≤50ms p95 (CORE-04, MVR item 3). Research measured the CURRENT baseline at 52–59ms p95, so a targeted jq-reduction in the shell gate is the expected (and only sanctioned) lever.

Purpose: CORE-09 (live reference probes both directions), CORE-04 (≤50ms p95 demonstrated, not asserted).
Output: `test_probe_runner.py`, `bench_recall.sh`, and (expected) a consolidated-jq `memory-recall.sh` shell gate with identical gate semantics.
</objective>

Behavior:
- Args: `-n <samples>` (default 20, the MVR minimum), `-p <payload-file>` (default: an embedded fire-path payload `{"tool_name":"Bash","tool_input":{"command":"nvidia-smi"},"cwd":"/tmp"}` — the fire path is the worst case and the honest gate measurement), optional `-s <store>` (default live store; the live ~144-memory catalog is the honest parse cost — a small fixture would flatter the number)

- Report: sort samples numerically, print `samples=<n>`, `p50_ms=<v>`, `p95_ms=<v>` (p95 = value at index ceil(0.95*n)-1 of the sorted list — with n=20 that is the 19th value), and `gate=PASS` if p95 ≤ 50 else `gate=FAIL`. Machine-parseable one-key-per-line so the MVR record can quote it.
